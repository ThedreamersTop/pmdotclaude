# 横纵分析法深度研究

> **方法论溯源**
> 横纵分析法由数字生命卡兹克（Khazix）提出，融合了语言学中的历时-共时分析（Saussure）、社会科学中的纵向-横截面研究设计、商学院案例研究法、以及竞争战略分析的核心思想，形成了一套适用于任意复杂研究对象（产品/公司/概念/方法/学科/技术/算法/开源项目/人物/事件/文化现象/科学话题）的通用研究框架。核心原则不变：纵向追时间深度，横向追同期广度，最终交汇出判断。

你正在执行一次横纵分析法深度研究。**先判断模式**：

- **深度模式**（默认）：10,000-30,000 字完整报告 + 排版精美的 PDF。适用于商业 due diligence、给团队的研究交付物、需要可分享/可归档的研究产出。
- **简版模式**：2,000-4,000 字 markdown，不生成 PDF。适用于个人快速 ramp up 一个陌生领域（学一篇论文背后的范式、了解一个新概念的脉络、一两个小时把一个领域装进脑子）。用户明确说"简版/快速/速读版/我只想快速搞懂"等任一信号时进入简版模式。

模式不同，搜索深度、报告篇幅、是否生成 PDF 都不同，但**方法论骨架（纵向 + 横向 + 交汇）保持一致**。下文中所有「篇幅 XX 字」的标注默认指深度模式；简版模式的对应篇幅在文末「简版模式」专节中给出。

## 前置准备

### 环境准备

1. **确认PDF转换脚本可用**：本Skill自带 `scripts/md_to_pdf.py`（基于WeasyPrint），用于将最终Markdown报告转为排版精美的PDF。确保依赖已安装：`pip install weasyprint markdown --break-system-packages`。
   - **CJK 字体依赖（Linux/devcontainer 必装）**：WeasyPrint 渲染中文需要系统装有 CJK 字体，否则中文会显示为豆腐字（□）。在 Debian/Ubuntu 系列：`sudo apt-get install -y fonts-noto-cjk`。macOS（已有 PingFang SC）和 Windows（已有 Microsoft YaHei）通常无需额外安装。脚本的 CSS 已经做了跨平台字体回退链 (Noto Sans CJK SC → PingFang SC → Microsoft YaHei → fallback)，确保字体已装就能跑。
2. **写作风格**：本Skill已内置完整的写作风格指南（见下文"写作风格"部分），无需额外加载其他skill。

### 明确研究对象

拿到用户输入后，确认以下信息。如果用户已经给得足够明确（比如"帮我用横纵分析法研究Hermes Agent"），不需要追问，直接开始：

1. **研究对象**：具体的产品名/公司名/概念名/方法名/技术名/算法名/开源项目名/人名/事件名/现象名
2. **对象类型**：从下表里选一个最贴近的——不同类型在词汇、信息来源、横向轴的定义上有显著差异，**类型判断错了，后面整个研究就会跑偏**。
3. **产出模式**：深度模式（默认）还是简版模式？深度模式跑 30-45 分钟出 PDF，简版模式跑 10-15 分钟出 markdown。
4. **研究动机**（可选）：为什么要研究它？最近发生了什么？
5. **特别关注点**（可选）：有没有特别想深入的方向？

### 对象类型速查表（**类型决定后续整个研究的"对照系"和"信息来源"**）

横纵分析法的方法论骨架是通用的，但具体的词汇和信息来源会随对象类型而变。**判断类型之后，下文所有出现"竞品""用户口碑""市场份额""创始团队"等业务词汇的地方，都要按下表替换。**

| 对象类型 | 横轴（"对照系"）含义 | "用户口碑"的对应 | "市场份额/影响力"的对应 | "创始团队"的对应 |
|---|---|---|---|---|
| **产品** | 同期竞品（功能/定价/体验维度） | G2/Capterra 评分、Reddit/Twitter 讨论、用户访谈 | 市场份额、ARR、用户数 | 产品团队、PM、关键工程师 |
| **公司** | 同赛道竞争公司 | 员工评价（Glassdoor）、客户证词、媒体口碑 | 营收、估值、市场份额、客户数 | 创始团队、关键早期员工 |
| **概念/方法/学科** | 同代对话者与论敌（**前辈/同辈/异邦/后辈** 四类） | 学术接受与争议事件、二手评论、教科书定位 | 引用频次、课纲覆盖度、学界自述 | 创始思想家及核心圈（如 Yale School、Tel Quel 小组） |
| **技术/算法/开源项目** | 同期技术路线、设计哲学不同的对照实现 | GitHub issues、HN/Reddit 讨论、技术博客、benchmark 论文 | GitHub stars 走势、生产采用率、引用次数、benchmark 排名 | 核心 committer、原始作者、关键贡献者 |
| **人物** | 同领域同代人物（路线对比） | 同行评价、传记、采访记录 | 影响力、引用、追随者、获奖 | 受谁影响、师承关系 |
| **历史事件** | 同期相关事件、平行事件、可比先例 | 当时舆论、亲历者记述、不同学派的历史解读 | 后续影响范围、引用频次 | 关键人物及决策圈 |
| **文化现象** | 同代/同地区的并行文化、上一波类似现象 | 参与者口述、社群讨论、文化评论 | 传播范围、商业化程度、模仿者数量 | 关键推动者、文化生产者 |
| **科学话题/疾病/药物** | 同类机制、同适应症对照、上一代解决方案 | 临床医生反馈、患者社区、循证医学评价 | 患者数、销售额、获批适应症数 | 发现者/PI/关键实验室 |

**横向轴的"对照系"四象限**（特别适用于概念/方法/学科/技术等抽象对象，作为"竞品"框架的扩展）：

- **前辈（predecessor）**：研究对象明确从中继承或反对的前一代方法/思想/技术。
- **同辈（contemporary）**：同时期、同问题域、不同进路的姊妹方法。
- **异邦（foreign tradition）**：在另一条传统/学科/文化里独立发展的、解决类似问题的方法。例：分析哲学之于解构、东亚禅宗之于现象学、CRDT 之于 OT。
- **后辈（successor）**：明确把研究对象当作"要超越的对象"建立自己身份的后续运动。

不是所有对象都有四类齐全，但能识别出几类，就能识别几类——比单纯列"竞品"信息量大得多。

### 信息来源的对象类型分支

下文「信息来源优先级」表的默认版本是产品/公司类的。若对象类型为其他，使用以下替代优先级：

- **概念/方法/学科**：Stanford Encyclopedia of Philosophy → Routledge Encyclopedia of Philosophy → Cambridge Companions → Notre Dame Philosophical Reviews → 同行评议期刊（Critical Inquiry、Diacritics、NLH 等）→ 学术专著 → academia.edu / PhilPapers preprints → Wikipedia（仅用于年表/参考书目）→ 通俗媒体。中文研究对象额外加：CNKI、中文学术专著、读书/天涯/豆瓣学术圈讨论。
- **技术/算法/开源项目**：原始 paper / RFC → 项目官方文档 → 项目仓库 issues/PRs/commits → benchmark 论文 → 关键贡献者博客/技术演讲 → HN / Lobste.rs / r/programming 讨论 → 二手技术博客。
- **人物**：传记（如有，且权威）→ 一手采访/口述 → 本人公开作品/发言 → 同行回忆录 → 学术评传 → 通俗传记/媒体报道。
- **历史事件**：当时一手史料（档案、报刊、亲历者日记/书信）→ 权威历史专著 → 不同学派的解读对比 → 通俗历史读物。
- **科学话题/疾病/药物**：PubMed / Cochrane systematic reviews → 临床指南（NICE、UpToDate、ESMO 等）→ NEJM / Lancet / JAMA 重要论文 → FDA/EMA 监管文件 → 患者社区 → 医疗新闻。
- **文化现象**：参与者口述/纪录片 → 当时社群发布物（forums、博客、社交媒体存档）→ 文化评论与学术研究 → 通俗媒体回顾。

**注意**：原表里"必查 arxiv"的指令仅适用于 AI / 算法 / CS / 物理 / 数学等以 arxiv 为主要预印本渠道的领域。**人文/哲学/历史/文化对象不要查 arxiv**，那不是它们的发表渠道。

## 第一步：联网信息收集

这个方法论的质量完全取决于信息的丰富度和准确性。**必须联网搜索**，不能仅靠已有知识。研究报告的价值在于深度和完整度，所以信息收集阶段宁可多搜，不要因为信息不够导致后面的分析浮于表面。

### 并行搜索策略

使用子Agent并行搜索来提高效率。**深度模式建议 2-3 个子 Agent，简版模式只用 1 个**。

**深度模式建议分工**（按对象类型调整词汇，业务对象用"竞品/用户口碑"，概念/技术等对象按上文「对象类型速查表」替换）：

- **子Agent 1 — 纵向信息**：研究对象的起源、创始人/创始思想家背景、发展历程、关键事件、版本/概念迭代、融资或学术接受、战略转向或思想转向、危机或重大争议事件
- **子Agent 2 — 横向信息**：对照系识别（业务对象=竞品；概念/方法=同代对话者与论敌的四类；技术=同期路线；以此类推）、各对照系的特点和评价、行业/学界对比、影响力指标
- **子Agent 3**（复杂对象才需要）：补充信息，如创始人/思想家深度背景、行业或学术环境变化、社群/用户/读者讨论（GitHub issues、Reddit、Twitter/X、知乎、学术评论网站等，按对象类型选择合适渠道）

**简版模式**：只派 1 个子 Agent，同时覆盖纵向+横向，但每个轴的搜索深度减半（target 10-15 分钟而非 30-45 分钟）。简版模式不需要补充信息子 Agent。

**派子 Agent 时务必把对象类型 + 模式明确传递**：在每个子 Agent 的 prompt 顶部声明 "对象类型：XX（产品/概念/技术/...），模式：YY（深度/简版）"，并附上对应的词汇和信息来源指引——否则子 Agent 会默认按产品/公司的业务词汇去搜，对学术/技术/人文对象会跑偏。

**子Agent联网工具使用指南**（直接写入每个子Agent的prompt中）：

每个子Agent的prompt中必须包含以下联网指引：

> 你需要联网获取信息。使用以下工具：
> - **WebSearch**：用于搜索发现信息来源，获取摘要和关键词结果
> - **WebFetch**：当已知具体URL时，用于从页面定向提取内容
> - 如果用户环境中安装了 web-access skill（检查路径 `/mnt/.claude/skills/web-access/SKILL.md` 是否存在），优先加载它并遵循其指引，它提供更强的浏览器CDP能力
> - 搜索策略：先用WebSearch发现信息来源和线索，找到具体URL后用WebFetch深入提取
> - 多次搜索、多个关键词组合，不要只搜一次就放弃
> - 一手来源优于二手来源：官方博客 > 权威媒体原创报道 > 转载/聚合
> - **学术类研究对象必查arxiv**：如果研究对象涉及学术概念、算法、AI模型、技术范式等，必须通过arxiv API获取相关论文。调用方式：`curl -s "https://export.arxiv.org/api/query?search_query=all:关键词1+AND+all:关键词2&max_results=10"`，或用WebFetch访问同一URL。返回XML格式，包含标题、作者、摘要、发布日期、PDF链接。可按需调整关键词组合和结果数量。找到关键论文后，用WebFetch读取论文页面（`https://arxiv.org/abs/论文ID`）获取更多细节。

prompt要描述目标（"获取""调研""了解"），不要用暗示具体手段的动词（"搜索""爬取"），让子Agent自主判断最佳获取方式。

### 信息来源优先级（默认版本：产品/公司类对象）

**对于非产品/公司对象，请优先参考上文「信息来源的对象类型分支」给出的替代优先级。**

一手来源优于二手来源，多个媒体引用同一个错误会造成循环印证假象：

| 信息类型 | 一手来源 |
|---------|---------|
| 产品更新/技术决策 | 官方博客、GitHub Release Notes、创始人推文 |
| 融资/商业数据 | 公司官方公告、SEC/工商文件 |
| 用户口碑 | G2/Capterra评分、GitHub Issues、Reddit讨论、Twitter/X、知乎帖子 |
| 行业分析 | 权威媒体原创报道（非转载） |
| 学术/技术原理 | arXiv论文（`export.arxiv.org/api/query`）、Google Scholar、学术会议论文集 |

### 信息充分性自检

搜索完成后检查：
- 纵向：能讲出一个完整的故事吗？有没有明显的信息断层？关键节点的决策动机有还原吗？
- 横向：对照系列表完整吗（产品=竞品；概念=同代对话者与论敌的四类；技术=同期路线）？有没有遗漏主要玩家？每个对照系的信息够做对比吗？
- 来源：关键事实有可靠来源支撑吗？有没有只靠单一来源就下判断的？对象类型选对了对应的源（不要给哲学概念查 arxiv，不要给算法只查 G2）？

信息不够就再补搜。不要凑合。

## 第二步：纵向分析（Diachronic / Longitudinal）

沿时间轴，完整还原研究对象从诞生到现在的发展全貌。这是报告的主体部分，篇幅应该最重。

### 内容要求

**起源追溯**：它诞生的背景是什么？基于什么技术/理念/需求而来？创始团队或核心推动者是谁？这些人之前做过什么，为什么是他们来做这件事？当时的行业环境是什么样的？有没有某个关键事件或灵感直接促成了它的诞生？

**诞生节点**：明确的首次发布/成立/提出时间，最初的形态和定位，跟现在有什么不同。

**演进历程**：从诞生到现在，按时间顺序梳理所有关键节点。包括但不限于：重大版本更新、融资事件、团队变动、战略转型、技术架构变化、用户规模里程碑、重大合作或收购、公关危机或争议事件。

**决策逻辑**：在每个关键节点上，尽可能还原决策背后的原因。为什么选了A而不是B？当时面对的约束条件是什么？哪些早期决策"锁定"了后来的发展方向、难以逆转？什么机制让它越走越深（网络效应、生态绑定、技术栈选择等）？

**阶段划分**：把整个历程自然分为几个阶段（萌芽期、快速增长期、转型期等），每个阶段有核心特征和核心矛盾。

### 篇幅

6000-15000字。历史越长、节点越多的对象靠近上限，新生事物靠近下限。核心原则是把故事讲完整、讲透，每个关键节点都值得展开，不要为了压缩而跳过重要细节。宁可写长写细，也不要蜻蜓点水。

## 第三步：横向分析（Synchronic / Cross-sectional）

以当前时间点为切面，将研究对象与**对照系**（按对象类型不同，可能是竞品、同代对话者、同期技术路线、平行人物等——见上文「对象类型速查表」）进行全面对比。

### 首先判断对照系情况

分三种场景处理：

**场景A：无直接对照。** 如果研究对象是全新品类、独占性极强的领域、或独一无二的现象，跳过逐一对比，改为分析：它为什么没有对照？是品类/方法太新、壁垒太高、还是问题域太独特？未来最可能从哪个方向冒出对照？有没有间接替代方案或上一代解决方式可以参照？历史上是否有结构相似但领域不同的可比先例？

**场景B：少量对照（1-2个）。** 逐一深入对比，每个对照系展开详细分析。

**场景C：对照充分（3个及以上）。** 选取最具代表性的 3-5 个进行对比，其余简要提及。**对概念/方法类对象**，强烈建议按「前辈/同辈/异邦/后辈」四象限组织对照系——这个分类比单纯并列竞品信息量大得多。

### 对比维度

根据研究对象的类型灵活调整，但至少覆盖以下方面（业务词汇在概念/技术等场景下要按「对象类型速查表」替换）：

**核心差异对比**：技术路线/核心方法论/底层逻辑、产品形态/商业模式/组织结构/思想立场、目标用户/受众/适用场景/适用问题域、核心优势与明显短板、定价策略/资源投入/规模体量/影响力指标。

**真实评价视角**：每个对照系的真实评价如何（产品=用户口碑；概念=学界接受与争议；技术=社群与生产采用反馈；人物=同行评价）？最常被提及的优点和短板分别是什么？官方定位 vs. 实际定位有没有偏差？对比不要写成参数对照表的文字版，要讲清楚每个对照系「活成了什么样 / 被读成了什么样 / 被用成了什么样」，受众选它的真实理由是什么。

**共同前提**（对概念/方法/技术类对象**特别重要**）：研究对象和各个对照系之间，有没有共享的底层假设/范式/前提？哪些争论是表面分歧，哪些是真正的根本分歧？这一步能防止"X vs Y"的懒惰对立框架。

**生态位分析**：在整个赛道/学界/技术栈/思想图谱中，研究对象占据什么位置？填补了什么空白，还是在跟谁正面竞争？当前格局是百花齐放、两强争霸、还是一家独大？

**代表性遭遇**（对概念/方法/技术类对象**强烈推荐**）：研究对象和主要对照系之间有没有真实发生过的论战、辩论、PK 时刻、协作或决裂事件？这些"遭遇瞬间"是横向分析最锋利的素材（比如解构 vs. 现象学之间有 Searle-Derrida 论战；分布式一致性算法之间有 Raft vs Paxos 论文）。

**趋势判断**：基于横向对比，研究对象在当前格局中的走向是什么？机会和风险各是什么？

### 篇幅

3000-10000字。场景A控制在3000字左右，场景C每个主要对照系（竞品/同代论敌/同期技术路线等，依对象类型）至少展开1500字以上的独立分析，不要一笔带过。

## 第四步：横纵交汇洞察

这是整篇报告的精华段。把纵向发展脉络和横向竞争格局结合起来，给出综合性的、新的判断。不要写成前面内容的缩写版。

需要回答的核心问题：

1. **历史如何塑造了当下的竞争位置**：纵向历程中的哪些决策和事件，决定了它今天在横向对比中的位置？
2. **对照系的纵向对比**：如果把主要对照系（竞品/同代论敌/同期技术路线等）也放到时间线上看，它们的起源和演变路径有什么不同？这些不同如何导致了今天各自的特点？
3. **优势的历史根源**：今天的每个核心优势，能追溯到历史上的哪个节点或决策？
4. **劣势的历史根源**：今天的每个核心劣势，能追溯到哪个历史决策？当初的「好决策」有没有变成今天的包袱？
5. **未来推演**：基于纵向趋势和横向竞争格局，给出三个剧本——最可能的、最危险的、最乐观的，每个剧本要有逻辑支撑。

### 篇幅

1500-3000字。

## 写作风格

这不是一份冷冰冰的咨询报告，而是一篇让人能从头读到尾的深度研究。写作风格需要在「研究报告的严谨」和「卡兹克的可读性」之间找到平衡点。

### 从卡兹克文风中借鉴的核心元素

以下风格元素直接应用到报告写作中（详细定义请参考 khazix-writer skill）：

**节奏感**：句子时长时短，段落之间跳跃自然。不要每段都一样长，一句话自成一段制造重量感的技巧可以用。好的节奏像波动，每次围绕主线偏出去一点，再用一句「扣主线句」拉回来。

**叙事驱动，不是罗列驱动**：纵向部分要有故事弧线，有起承转合。比如一个产品为什么在某个时间点突然爆发，背后的铺垫是什么，转折是什么。不要写成"2023年1月发布了A，2023年3月发布了B"这种流水账。

**知识是「聊着聊着顺手掏出来」的**：在讲述过程中自然地带出背景知识，不要「下面我来给大家科普一下」。

**敢下判断**：鼓励给出观点和洞察，但每个观点必须有事实支撑。先摆事实，再给判断。是推测的明确标注。表达判断时用「我觉得」「我的判断是」这种承认主观性的姿态，而不是居高临下的定论。

**层层剥开的修辞**：不直接讲结论，用"现象→表面解释→更深的追问→核心洞察"的方式展开。让读者参与到思考过程中。

**文化升维**：在交汇洞察部分，连接到更大的文化/哲学/历史参照物。不是硬凑的升华，是「聊着聊着自然想到了」的感觉。

**回环呼应**：开头或纵向部分埋的细节和钩子，在交汇洞察或结尾callback回来。前后因果的闭合感，是让报告从「信息流」变成「作品」的关键。

### 不从卡兹克文风中借鉴的元素

以下元素适合公众号文章但不适合研究报告，需要克制：

- **过强的口语化**：报告可以有聊天感，但不要满篇「这玩意」「不是哥们」「太牛逼了」。偶尔点缀可以，但密度要比公众号文章低很多。
- **去小标题化**：公众号文章追求一口气顺下来不加小标题。研究报告不一样，1-3万字的内容如果没有清晰的结构和导航，读者会迷路。报告需要清晰的章节结构。
- **标点禁令可以放松**：公众号文章禁用冒号和破折号。研究报告中可以正常使用，因为报告需要的是信息传达效率。但「」的使用习惯可以保留。
- **固定尾部**：不要加公众号的三连/星标尾部。

### 绝对禁区（依然适用）

以下AI味标记无论什么文体都要避免：
- 套话："首先...其次...最后"、"综上所述"、"值得注意的是"、"不难发现"
- 空洞形容词："赋能"、"抓手"、"打造闭环"
- 教科书开头："在当今AI快速发展的时代"、"随着技术的不断进步"
- 高频踩雷词："说白了"、"意味着什么？"、"这意味着"、"本质上"、"换句话说"、"不可否认"
- 空泛工具名：不说"AI工具"、"某个模型"，要说具体名字
- 编造场景：如果某个信息搜不到，诚实标注「该信息暂缺」，绝不编造

### 用人话写

避免咨询公司式的套话和空洞概括。用具体的细节和例子代替概括性陈述。比如不要写「该公司在这一阶段实现了快速增长」，而要写「从2024年中期的1000万美元ARR到2025年底的10亿美元，增长曲线几乎是垂直的」。

## 不同研究对象类型的适配

核心原则不变（纵向追时间深度，横向追同期广度），但侧重点不同。下面给出 8 种典型对象类型的具体适配——**判断对象类型之后，认真读对应那段**：

**研究产品时**：纵轴重点关注版本迭代、技术路线演变、用户增长曲线、关键产品决策、商业模式调整、与平台/生态的关系变化；横轴重点关注功能对比、性能对比、用户体验、定价、目标用户细分差异。

**研究公司时**：纵轴重点关注创始团队、融资历程、战略转向、组织变革、关键人事变动、商业模式迭代；横轴重点关注商业模式差异、市场份额、营收对比、组织架构差异、企业文化差异。

**研究概念/方法/学派时**（哲学概念、学术范式、研究方法、思想流派）：纵轴重点关注概念的**思想前史**（基于什么前辈传统、回应什么问题）、**创始时刻**（谁、何时、何地、通过哪几个关键文本提出）、**关键术语的出场顺序**（每个核心概念是哪一年由哪个文本引入）、**扩散到其他学科的路径**、**重大争议与遭遇事件**（论战、丑闻、辩论会）、**晚期阶段或转向**、**逝世后/退潮后的接受与反弹**、**非西方/非主流场域的接受史**。横轴用"对照系四象限"组织：**前辈**（明确继承或反对的前代思想/方法）、**同辈**（同代姊妹方法）、**异邦**（另一传统里解决类似问题的方法，如分析哲学之于解构、东亚思想之于现象学）、**后辈**（明确把研究对象当作要超越对象建立自己身份的后续运动）。**关键提示**：用"代表性遭遇事件"（具体的一次论战、一篇回应文章、一场会议）作为对比的锚，远胜于抽象比较。

**研究技术/算法/开源项目时**：

- **必做第 0 步：明确问题域**。技术对象常跨多个问题域（如 CRDT 既是分布式一致性问题也是协作编辑问题；Raft 既属于共识算法也属于状态机复制）。研究启动前先和用户/上下文确认"研究的是哪个问题域下的它"——否则纵向叙事会在不同应用场景之间反复跳跃。
- 纵轴重点关注**核心设计灵感来源**（哪些前代技术/论文/项目）、**首次发布**（版本号、commit、paper 投稿）、**关键设计决策的取舍**（CAP、性能 vs 一致性、易用性 vs 灵活性等）、**重大版本变更与破坏性变化**、**生态扩展**（语言绑定、衍生项目）、**性能 benchmark 的演变**、**社区治理与冲突**（fork、争议、license 变更）。
- 横轴重点关注**同问题域的不同设计哲学**（如 Raft vs Paxos vs Multi-Paxos、PostgreSQL vs MySQL vs SQLite vs DuckDB、React vs Vue vs Svelte、CRDT vs OT 不只是"哪个更好"，而是"它们各自在赌什么 trade-off"），还有**生产采用率分布**（用在什么规模/什么类型的公司、什么场景下被替换、什么场景下不可替代）。
- **强烈推荐用"代表性遭遇事件"作为对比锚**——技术领域的遭遇事件（fork、license 变更、benchmark 论战、关键人物公开倒戈，例如 Joseph Gentle 2022 年从 OT 阵营公开倒向 CRDT）信息密度极高，比抽象的"X vs Y 优缺点对比"强十倍。
- **四象限归类规则**：技术对象的"前辈/同辈/异邦/后辈"经常重叠（OT 既是 CRDT 的前辈也是同辈；Paxos 之于 CRDT 既是前辈也是异邦）。**按对研究对象的影响关系归类，不按时间先后**——比如 OT 主要起"被对照"和"路线之争对手"作用，归为同辈；Paxos 主要起"基础参照"作用，归为前辈。
- **交汇部分必答**：如果存在**路线之争**（CRDT vs OT、SQL vs NoSQL、Raft vs Paxos、REST vs GraphQL 等），必须在横纵交汇部分给出"这场路线之争当前打到哪一步"的判断。不要回避，不要"两者各有优劣"地骑墙——明确给出比分判断和最近的转折性事件。

**研究人物时**：纵轴重点关注**家庭/教育背景**、**师承关系**、职业轨迹、关键决策、成长曲线、思想/风格演变、公开言论变化、与同代人的关系网；横轴重点关注与同领域其他人物的对比（做事方式、风格、成就、影响力、路线选择差异），以及**师承支系的对比**（同一个老师的不同弟子如何走向不同方向）。

**研究历史事件时**：纵轴重点关注**事件的远因与近因**（结构性条件 vs 触发因素）、**关键时间节点的决策**、**当时各方反应的演变**、**事件后果的展开**（短期、中期、长期）、**历史评价的变迁**（同一事件被不同年代不同立场重新解读）；横轴重点关注**平行事件**（同期其他地区类似事件）、**可比先例**（结构相似的早期事件）、**对照决策**（在同样困境下其他人选择了什么不同路径）。

**研究文化现象时**（流行运动、亚文化、网红现象、艺术运动）：纵轴重点关注**萌芽期的微小信号**（早期参与者、最初的载体）、**扩散机制**（哪些媒介把它放大）、**商业化或反商业化阶段**、**主流化或反主流化阶段**、**衰退或转变**；横轴重点关注**同代的并行现象**（往往是同样的代际情绪在不同地区/媒介上的不同表达）、**上一波类似现象**（结构相似的历史先例）、**对立面**（与之竞争同一代人注意力的反向现象）。

**研究科学话题/疾病/药物时**：纵轴重点关注**问题被识别的时刻**（谁、何时、在什么病例/实验中第一次提出）、**机制理解的演变**（每一代的主流假说和其失败）、**关键临床试验或决定性实验**、**监管批准/指南更新的节点**、**社会层面的认知转变**（疾病被去污名化、新一代药物改变病患生活质量等）；横轴重点关注**同类机制**（同靶点其他药物、同发病机制其他疾病）、**同适应症对照**（治疗同一疾病的不同方案，对照各自有效率、副作用、可及性）、**上一代解决方案**（被研究对象替代的旧标准是什么、为什么被替代）。

**所有对象类型通用**：纵轴永远要回答"它怎么变成今天这个样子的"，横轴永远要回答"它周围还有什么、它在这个景观里占什么位置"，交汇永远要回答"今天的样子和当下的位置之间有什么因果关系"。词汇会变，结构不变。

## 简版模式（Brief Mode）

**何时使用**：用户想在 1-2 小时内 ramp up 一个陌生领域，把它装进自己的工作记忆里。不需要交付物，不需要分享给别人。典型触发：「帮我快速了解 XX」「我想搞懂 XX 这个领域，简版就行」「速读版」「10 分钟版」「不用 PDF」。

**与深度模式的差别**：

| 维度 | 深度模式（默认） | 简版模式 |
|---|---|---|
| 子 Agent 数量 | 2-3 个并行 | 1 个综合（或主 Agent 直接并行 WebSearch/WebFetch） |
| 单次搜索时长目标 | 30-45 分钟 | 10-15 分钟 |
| 纵向篇幅 | 6,000-15,000 字 | 1,100-1,500 字 |
| 横向篇幅 | 3,000-10,000 字 | 700-1,000 字 |
| 交汇洞察篇幅 | 1,500-3,000 字 | 500-800 字 |
| **总字数** | **10,000-30,000 字** | **2,500-3,500 字**（实测自然落点，技术对象偏上沿，概念对象偏下沿） |
| 产出格式 | Markdown + PDF | 仅 Markdown |
| 排版/封面 | 需要 | 不需要 |
| 写作风格 | 完整卡兹克叙事风格 | 同样的风格但密度更高、节奏更快 |
| 信息来源 | 全列引用清单 | 5-10 个关键来源即可 |

**简版模式的核心原则**：
1. **不省方法论骨架**：纵 + 横 + 交汇三轴依然完整，只是每轴篇幅压缩。一上来就只写纵向不写横向，是错的——会丢掉横纵分析法的核心价值。
2. **优先讲叙事而非堆事实**：简版字数少，更要靠故事结构和因果链条让读者一口气读完。流水账列举是简版的死敌。
3. **每个对照系一段就够**：横向部分不展开每个对照系，每个只用一段（150-300 字）讲清楚它跟研究对象的关系和共同前提，剩下交给读者自己延伸。
4. **⚠️ 交汇部分绝对不能省**：哪怕只剩 500 字，也**必须**把"历史如何塑造了当下"这个判断给出来——这是读者花一小时换来的**核心收益**。简版写作过程中若发现字数吃紧、想砍内容，**永远先砍纵向细节、永远不砍交汇**。
5. **不生成 PDF**：直接输出 markdown 到当前工作目录，文件名 `[研究对象]_横纵速读.md`。
6. **判断 vs 事实陈述的措辞区分**：「我觉得」「我的判断是」这类表态留给**终极判断**（横纵交汇里的核心结论）使用，**不要**散布在事实陈述里。事实陈述用直接断言（"X 发生于 2014 年"），不要写成"我觉得 X 发生于 2014 年"——后者既绕又稀释了真正判断的份量。

**简版模式的子 Agent prompt 模板**（单个综合 Agent）：

> 你需要做一次横纵分析法的**简版**研究，目标是让用户用 10-15 分钟读完后能把这个领域装进工作记忆里。
> 对象：[研究对象]
> 对象类型：[从对象类型速查表选一个]
> 在你的搜索里，平均分配时间给纵向和横向。纵向要找出：思想/技术/产品的源头 + 3-5 个关键节点 + 当前状态。横向要找出：3-5 个对照系（[根据对象类型给出对应词汇，如同代论敌/同期技术路线/竞品]）+ 它们各自的核心立场。
> 不需要找完整的引用清单，每个关键事实有一个可靠来源即可。**信息来源优先级**：[根据对象类型给出对应来源，如 SEP/Wikipedia/官方文档/原始 paper]。

**简版模式的报告结构模板**：

```
# [研究对象] · 横纵速读

> 研究时间：YYYY-MM-DD | 类型：[对象类型] | 模式：简版

## 一句话定义
[80-150 字]

## 纵向：从源头到当下
[1,200-2,000 字，分 3-5 个段落或小节，每段对应一个关键阶段]

## 横向：周边的对照系
[600-1,200 字，每个主要对照系一段，重点讲共同前提和真正的分歧]

## 横纵交汇
[400-800 字，给出 1-2 个主判断和"为什么会变成今天这样"的因果链]

## 关键来源
[5-10 个 URL，足够读者自己深入即可]
```

简版模式适合个人快速 ramp up；如果之后想转成深度版本，直接用同一个研究对象重新启动深度模式即可——前面的简版材料是很好的预热。

## 篇幅总览（深度模式）

| 部分 | 字数范围 | 说明 |
|-----|---------|------|
| 纵向分析 | 6,000 - 15,000字 | 报告主体，不要蜻蜓点水 |
| 横向分析 | 3,000 - 10,000字 | 视对照系数量调整 |
| 横纵交汇 | 1,500 - 3,000字 | 精华段，给出新判断 |
| **全文总计** | **10,000 - 30,000字** | 不要怕长，深度和完整度是价值所在 |

简版模式的篇幅参考上文「简版模式」节中的对照表。

## 质检清单

### 通用质检（深度模式 + 简版模式都要过）

- [ ] **对象类型判断正确**？纵向/横向/交汇所用的词汇和信息来源跟对象类型匹配（没有给哲学概念查 arxiv，没有给算法只查 G2 评分）？
- [ ] 纵轴是叙事故事体？读起来有因果逻辑和时代脉络？不是年表流水账？
- [ ] 创始人/发起者/创始思想家的背景和动机有足够深度？
- [ ] 每个关键节点都展开写了，没有为了压缩而跳过重要细节（深度模式）/没有为了篇幅而跳过最关键的 3-5 个节点（简版模式）？
- [ ] 决策/思想转向的**逻辑**有还原？不只是「发生了什么」，还有「为什么这么选」？
- [ ] 横轴的对照系场景判断正确（A/B/C）？对照系分析够深？**对概念/方法类对象，是否尝试用前辈/同辈/异邦/后辈四象限组织对照系**？
- [ ] **共同前提**有讨论吗？还是只写了"X vs Y"的对立面？真正的分歧和表面的分歧区分了吗？
- [ ] 真实评价/口碑部分引用了真实声音（产品=用户、概念=学者、技术=社群、人物=同行），不只是官方宣传或自我标榜？
- [ ] 横纵交汇产出了**新的**判断，不是前面内容的缩写版？
- [ ] 未来推演的剧本（深度模式三个，简版模式至少一个）都有逻辑支撑？
- [ ] 写作风格有节奏感、有可读性？不是冷冰冰的咨询报告？
- [ ] 没有触犯绝对禁区里的任何一条？
- [ ] 所有关键事实标注了信息来源？
- [ ] 搜不到的信息诚实标注了「暂缺」，没有编造？

### 深度模式专属质检

- [ ] PDF 排版美观、结构清晰、可读性好？
- [ ] 总字数在 10,000-30,000 字的范围内？
- [ ] 信息来源章节完整列出了所有引用 URL？

### 简版模式专属质检

- [ ] 总字数在 2,000-4,000 字之间？（明显超出说明应该走深度模式而非简版）
- [ ] 没有为了"显得完整"而在简版里塞进深度模式的 boilerplate？
- [ ] 5-10 个关键来源足够让读者自己进一步深入？
